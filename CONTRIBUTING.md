# Contributing

Thanks for helping improve animepahe-downloader. This guide covers local setup,
the build/test commands, and the conventions a pull request must follow.

## Development setup

- **Go** — the project targets Go 1.21+. In this repo's environment Go lives at
  `$HOME/.local/go` and is **not on `PATH` by default**, so either prepend it:

  ```sh
  export PATH=$HOME/.local/go/bin:$PATH
  ```

  or use the `Makefile`, which prepends it for you.

- **ffmpeg** — required on `PATH` to actually download (the working path is HLS).
  Without it the build and tests still run, and `--export` still works, but real
  downloads fail with a clear message.

## Build, test, vet

Using the `Makefile` (recommended — it handles the Go `PATH`):

```sh
make build         # compile the ./apahe binary
make test          # run unit tests
make vet           # go vet
make check         # fmt + vet + test — the pre-commit gate
```

Or directly (remember to put Go on `PATH` first):

```sh
go build -o apahe .
go vet ./...
go test ./...
go test ./internal/kwik/ -run TestUnpack -v   # a single test
```

Run `make check` before opening a pull request.

## Commit messages

This project uses [Conventional Commits](https://www.conventionalcommits.org/)
and they are **required**. The release changelog is generated by goreleaser,
which filters commits by their type prefix (`docs:`, `test:`, `ci:` are excluded
from release notes). Messages that don't follow the convention pollute the
published changelog.

Use a type prefix such as `feat:`, `fix:`, `docs:`, `test:`, `ci:`, `refactor:`,
or `chore:`, for example:

```
feat: add --output flag to set download directory
fix: handle 302 redirect without trailing slash
```

## Pull request flow

1. Branch off `main`.
2. Make your change; keep it focused.
3. Ensure `make check` is green (fmt, vet, tests pass).
4. Open a pull request describing the change and the motivation.

CI runs vet, build, tests, and a goreleaser dry-run on every pull request — all
must pass.
